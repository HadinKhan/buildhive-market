## 🎯 Project Overview

**BuildHive Marketplace** is a B2B/B2C e-commerce platform designed specifically for Pakistan's construction industry. It enables:

- **Buyers/Contractors**: Browse and purchase construction materials
- **Suppliers**: List and manage products
- **Admins**: Oversee the entire marketplace

The frontend is a **Single Page Application (SPA)** that communicates with a backend API (Supabase) for data persistence and authentication.

### 2. `vite.config.ts` - Build Tool Configuration

**What it does**: Configures Vite's dev server, build process, and environment variables.

```typescript
import { defineConfig, loadEnv } from "vite";
import react from "@vitejs/plugin-react";

export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, ".", ""); // Load .env.local variables
  return {
    server: {
      port: 4200, // Dev server runs on port 4200
      host: "0.0.0.0", // Allow external access (mobile testing)
    },
    plugins: [react()], // Enable React Fast Refresh
    define: {
      // Inject env variables into client code
      "process.env.GEMINI_API_KEY": JSON.stringify(env.GEMINI_API_KEY),
    },
    resolve: {
      alias: {
        "@": path.resolve(__dirname, "."), // '@' alias for imports
      },
    },
  };
});
```

**Why This Configuration?**:

- **Port 4200**: Custom port to avoid conflicts (default is 5173)
- **host: '0.0.0.0'**: Allows testing on mobile devices via LAN IP
- **loadEnv()**: Loads environment variables from `.env.local` securely
- **define**: Replaces `process.env.GEMINI_API_KEY` at build time (for AI features)
- **@/ alias**: Write `import X from '@/components/X'` instead of `'../../components/X'`

### 3. `tsconfig.json` - TypeScript Configuration

**What it does**: Tells TypeScript how to compile and check your code.

```json
{
  "compilerOptions": {
    "target": "ES2022", // Compile to modern JavaScript
    "module": "ESNext", // Use ES modules (import/export)
    "lib": ["ES2022", "DOM"], // Include browser APIs
    "jsx": "react-jsx", // Use new JSX transform (no import React)
    "moduleResolution": "bundler", // Vite handles module resolution
    "skipLibCheck": true, // Skip type checking of .d.ts files (faster)
    "allowImportingTsExtensions": true, // Allow .tsx imports
    "noEmit": true, // Don't emit JS (Vite handles this)
    "paths": {
      "@/*": ["./*"] // @ alias for absolute imports
    }
  }
}
```

**Why These Settings?**:

- **target: ES2022**: Modern browsers support this (async/await, optional chaining)
- **jsx: react-jsx**: React 17+ doesn't require `import React` in every file
- **moduleResolution: bundler**: Optimized for Vite (not Node.js)
- **noEmit: true**: TypeScript only checks types; Vite compiles to JS
- **paths**: Matches Vite's @ alias for consistency

### 2. `App.tsx` - Routing & Layout

**What it does**: Defines all routes and wraps pages in shared layout (Header/Footer).

```typescript
import { Routes, Route, Navigate } from "react-router-dom";
import { Header, Footer } from "./components/Layout";
import { useAuth } from "./src/context/AuthContext";

// Protected route wrapper
const ProtectedRoute = ({ children, redirectTo = "/signin" }) => {
  const { isAuthenticated } = useAuth();
  return isAuthenticated ? (
    <>{children}</>
  ) : (
    <Navigate to={redirectTo} replace />
  );
};

function App() {
  const [cart, setCart] = useState<CartItem[]>([]);

  return (
    <div className="flex flex-col min-h-screen">
      <Header cartItems={cart} /> {/* Always visible */}
      <main className="flex-grow">
        <Routes>
          <Route path="/" element={<HomePage />} />
          <Route path="/products" element={<ProductsPage />} />
          <Route path="/products/:id" element={<ProductDetailPage />} />
          <Route path="/cart" element={<CartPage />} />

          {/* Protected routes */}
          <Route
            path="/checkout"
            element={
              <ProtectedRoute>
                <CheckoutPage />
              </ProtectedRoute>
            }
          />

          <Route
            path="/account"
            element={
              <ProtectedRoute>
                <AccountPage />
              </ProtectedRoute>
            }
          />

          {/* ... more routes */}
        </Routes>
      </main>
      <Footer /> {/* Always visible */}
      <ToastContainer /> {/* For notifications */}
    </div>
  );
}
```

**Why This Pattern?**:

- **Flex layout**: `flex-grow` makes main content fill space between header/footer
- **ProtectedRoute**: Redirects unauthenticated users to /signin automatically
- **Shared state**: Cart is managed in App and passed to components
- **ToastContainer**: Renders all toast notifications at root level

### 3. `components/Button.tsx` - Reusable Button

**What it does**: Provides a consistent button component with multiple variants.

```typescript
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: "primary" | "outline" | "ghost" | "gradient";
  size?: "sm" | "md" | "lg";
}

export const Button = ({
  variant = "primary",
  size = "md",
  className,
  ...props
}) => {
  const variants = {
    primary: "bg-primary text-white hover:bg-primary-hover",
    outline: "border border-gray-200 bg-white text-gray-700 hover:bg-gray-50",
    ghost: "bg-transparent text-gray-600 hover:text-primary",
    gradient: "bg-gradient-to-r from-violet-500 to-fuchsia-500 text-white",
  };

  const sizes = {
    sm: "px-4 py-1.5 text-xs",
    md: "px-6 py-2.5 text-sm",
    lg: "px-8 py-3 text-base",
  };

  return (
    <button
      className={`inline-flex items-center justify-center rounded-full font-medium 
                  transition-all focus:ring-2 ${variants[variant]} ${sizes[size]} ${className}`}
      {...props}
    />
  );
};
```

**Why This Approach?**:

- **Single source of truth**: All buttons look consistent
- **Flexible**: Pass any HTML button attribute (onClick, disabled, type, etc.)
- **Composable**: Can override className for special cases
- **Accessible**: Focus ring for keyboard navigation

**Usage**:


```typescript
const [cart, setCart] = useState<CartItem[]>([]);

const addToCart = (product: Product, quantity: number) => {
  setCart((prev) => {
    const existing = prev.find((item) => item.id === product.id);
    if (existing) {
      return prev.map((item) =>
        item.id === product.id
          ? { ...item, quantity: item.quantity + quantity }
          : item
      );

    return [...prev, { ...product, quantity }];

};
```

## 🔌 Services Layer

**What is it?**: Abstraction layer between React components and backend API.

**Why Separate Services?**:

- **Separation of concerns**: Components don't know API details
- **Reusability**: Multiple components use same service
- **Testing**: Easy to mock services in tests
- **Consistency**: All API calls follow same pattern

---

### 1. `src/services/api.ts` - Base HTTP Client

**What it does**: Creates configured Axios instance.

```typescript
import axios from "axios";

const api = axios.create({
  baseURL: import.meta.env.VITE_API_URL, // From .env.local
  timeout: 30000, // 30 second timeout
  headers: {
    "Content-Type": "application/json",
  },
});

// Request interceptor (add auth token to all requests)
api.interceptors.request.use((config) => {
  const token = localStorage.getItem("authToken");
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// Response interceptor (handle 401 errors)
api.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem("authToken");
      window.location.href = "/signin"; // Redirect to login
    }
    return Promise.reject(error);
  }
);

export default api;
```

**Why Interceptors?**:

- **Auto-attach token**: Don't need to manually add header in every request
- **Global error handling**: 401 errors auto-logout user
- **Logging**: Can add console.log for debugging all API calls

### 2. `productService.ts` - Product Operations

**Methods**:

```typescript
export const productService = {
  // Get all products with filters
  async getProducts(params?: {
    search?: string;
    category_id?: string;
    page?: number;
    limit?: number;
    status?: string;
  }) {
    const response = await api.get("/products", { params });
    return response.data;
  },

  // Get single product
  async getProductById(id: string) {
    const response = await api.get(`/products/${id}`);
    return response.data;
  },

  // Create product (supplier only)
  async createProduct(data: CreateProductDTO) {
    const response = await api.post("/products", data);
    return response.data;
  },

  // Update product
  async updateProduct(id: string, data: Partial<Product>) {
    const response = await api.put(`/products/${id}`, data);
    return response.data;
  },

  // Delete product
  async deleteProduct(id: string) {
    await api.delete(`/products/${id}`);
  },
};
```

**Why This Pattern?**:

- **Type safety**: TypeScript knows what data to expect
- **Single responsibility**: productService only handles products
- **Easy to mock**: In tests, replace with fake data

```typescript
    const response = await api.post("/auth/signin", { email, password });
    const { user, token } = response.data;
    localStorage.setItem("authToken", token);
    return user;
  },

  async signUp(data: { email; password; full_name; role }) {
    const response = await api.post("/auth/signup", data);
    return response.data;
  },

  async signOut() {
    localStorage.removeItem("authToken");
    await api.post("/auth/signout");
  },

  async getCurrentUser() {
    const response = await api.get("/auth/me");
    return response.data;
  },
};
```

**Token Storage**:

- **localStorage**: Persists across page refreshes
- **Alternative**: sessionStorage (cleared when tab closes)
- **Why not cookies?**: More complex (CSRF protection needed)

## 🌐 State Management

### 1. React Context (AuthContext)

**What it does**: Shares authentication state across all components.

```typescript
interface AuthContextType {
  user: User | null;
  isAuthenticated: boolean;
  signIn: (email: string, password: string) => Promise<void>;
  signUp: (data: SignUpData) => Promise<void>;
  signOut: () => void;
  loading: boolean;
}

export const AuthContext = createContext<AuthContextType>(null!);

export const AuthProvider = ({ children }) => {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);

  // Check if user is logged in on mount
  useEffect(() => {
    const token = localStorage.getItem("authToken");
    if (token) {
      authService
        .getCurrentUser()
        .then(setUser)
        .catch(() => localStorage.removeItem("authToken"))
        .finally(() => setLoading(false));
    } else {
      setLoading(false);
    }
  }, []);

  const signIn = async (email, password) => {
    const user = await authService.signIn(email, password);
    setUser(user);
  };

  const signOut = () => {
    authService.signOut();
    setUser(null);
  };

  return (
    <AuthContext.Provider
      value={{ user, isAuthenticated: !!user, signIn, signOut, loading }}
    >
      {children}
    </AuthContext.Provider>
  );
};

// Custom hook for easy access
export const useAuth = () => useContext(AuthContext);
```

**Usage in Components**:

```typescript
function ProfileButton() {
  const { user, isAuthenticated, signOut } = useAuth();

  if (!isAuthenticated) {
    return <Link to="/signin">Sign In</Link>;
  }

  return (
    <div>
      <span>Hello, {user.full_name}</span>
      <button onClick={signOut}>Logout</button>
    </div>
  );
}
```

**Why Context vs Redux?**:

- **Simpler**: No boilerplate (actions, reducers, store)
- **Built-in**: No external library needed
- **Sufficient**: Auth state is simple (user object + token)
- **When Redux?**: If state becomes complex (100+ actions, time-travel debugging needed)

## 🚀 Routing Architecture

**Why React Router?**:

- **SPA Navigation**: Change pages without full reload
- **URL Parameters**: `/products/:id` → `params.id`
- **Query Strings**: `/products?search=cement` → `searchParams.get('search')`
- **Protected Routes**: Redirect unauthorized users
- **Nested Layouts**: Share header/footer across pages

**Route Types**:

1. **Public Routes** (anyone can access):

```typescript
<Route path="/" element={<HomePage />} />
<Route path="/products" element={<ProductsPage />} />
<Route path="/signin" element={<SignInPage />} />
```

2. **Protected Routes** (auth required):

```typescript
<Route
  path="/account"
  element={
    <ProtectedRoute>
      <AccountPage />
    </ProtectedRoute>
  }
/>
```

3. **Dynamic Routes** (URL parameters):

```typescript
<Route path="/products/:id" element={<ProductDetailPage />} />
// Matches /products/123, /products/abc, etc.
```

**Navigation Methods**:

1. **Declarative** (JSX):

```typescript
import { Link } from "react-router-dom";

<Link to="/products">Browse Products</Link>;
```

2. **Programmatic** (JavaScript):

```typescript
import { useNavigate } from "react-router-dom";

function LoginForm() {
  const navigate = useNavigate();

  const handleSubmit = async () => {
    await authService.signIn(email, password);
    navigate("/account"); // Redirect after login
  };
}
```

3. **URL State** (persist filters):

```typescript
import { useSearchParams } from "react-router-dom";

const [searchParams, setSearchParams] = useSearchParams();

// Read: ?page=2
const page = searchParams.get("page");

// Write: ?page=3
setSearchParams({ page: "3" });
```

## 🔄 How Data Flows

### Example: Adding Product to Cart

1. **User clicks "Add to Cart"** in ProductDetailPage
2. **Event handler calls** `onAddToCart(product, quantity)`
3. **App.tsx receives** call, updates cart state
4. **Cart state triggers re-render** of Header (cart badge updates)
5. **Toast notification** shows "Added to cart!"

```
ProductDetailPage
    │
    │ onClick
    ├──> onAddToCart(product, 2)
    │
App.tsx
    │
    │ setCart([...cart, { product, quantity: 2 }])
    │
    ├──> Header (cart badge: 1 → 3)
    └──> toast.success("Added to cart!")
```

### Example: Fetching Products with Filters

1. **User types in search box** on ProductsPage
2. **Debounced input** waits 500ms after typing stops
3. **setSearchParams** updates URL (`?search=cement`)
4. **useEffect triggers** (dependency: searchParams)
5. **API call** `productService.getProducts({ search: 'cement' })`
6. **Backend returns** filtered products
7. **setProducts** updates state, triggers re-render
8. **ProductGrid** displays new products

```
User types "cement"
    │
    │ (500ms debounce)
    ├──> setSearchParams({ search: 'cement' })
    │
useEffect (dependency: searchParams)
    │
    ├──> productService.getProducts({ search: 'cement' })
    │
API Request
    │
    │ GET /products?search=cement
    │
Backend responds with products[]
    │
    ├──> setProducts([...])
    │
ProductGrid re-renders with new products
```

## 🎯 Key Design Decisions

### 1. Why Vite instead of Create React App?

- **CRA is deprecated** (React team no longer maintains)
- **Vite is 10-100x faster**: Uses native ES modules
- **Instant HMR**: Changes appear immediately
- **Smaller bundles**: Better tree-shaking

### 2. Why TypeScript instead of JavaScript?

- **Catch bugs early**: Type errors before runtime
- **Better autocomplete**: IDE knows what properties exist
- **Self-documenting**: Types show what functions expect
- **Safer refactoring**: TS catches breaking changes

### 3. Why React Router instead of manual routing?

- **Standard**: Industry-standard solution
- **Features**: Nested routes, lazy loading, code splitting
- **URL management**: Query params, history, redirects
- **Mobile-ready**: Handles back button correctly

### 4. Why Axios instead of Fetch?

- **Interceptors**: Auto-add auth token, handle 401 errors globally
- **Timeouts**: Built-in timeout support
- **Better errors**: Automatic JSON parsing
- **Less boilerplate**: Simpler error handling

### 5. Why Context instead of Redux?

- **Simpler**: No actions, reducers, store setup
- **Built-in**: No external library
- **Sufficient**: Auth state is simple
- **Later**: Can migrate to Redux if state grows complex

### 6. Why Services layer?

- **Separation of concerns**: Components don't know API details
- **Reusability**: Multiple components use same service
- **Testing**: Easy to mock services
- **Maintainability**: Change API without touching components

---

## 🔒 Security Considerations

1. **JWT Tokens**: Stored in localStorage (XSS risk - consider httpOnly cookies later)
2. **HTTPS**: Always use HTTPS in production (prevents token theft)
3. **Input Validation**: Always validate on backend (frontend validation is for UX)
4. **SQL Injection**: Backend uses parameterized queries (Supabase handles this)
5. **CORS**: Backend must whitelist frontend domain
6. **Environment Variables**: Never commit .env.local (in .gitignore)
